# Triage: Fallback to Full-File Read and Policy-Mapped Response

When initial grep or targeted searches return no matches during a support-ticket triage task, escalate immediately to a full `Read` of the relevant policy or knowledge file (e.g., `policies.md`). Do not guess or proceed without complete context. This fallback ensures accurate, non-hallucinated answers even when search patterns fail.

When extracting policy information to answer a triage request, map each recommended action explicitly to the specific policy section that authorises it. Present this as a structured table or clearly labelled list so citations are auditable and traceable. Avoid stating policy rules with confidence unless you have verified them against the source file in the current session.

When the policy supports multiple valid resolution paths (e.g., replacement vs. refund), present both options with their distinctions clearly noted. Frame the response to give the handling agent flexibility without overstepping documented boundaries.

If a real-world requirement (such as requesting proof of damage before processing a claim) is not mandated by the policy document as written, do not add it as a requirement. Only enforce what the policy explicitly states; note optional best-practice steps as suggestions only, not obligations.

A blank or low-confidence refining does not block a correct response. When the pre-turn plan is absent, self-direct by reading available source files, extracting relevant sections, and structuring the output as: recommended actions → policy citations → escalation check.
